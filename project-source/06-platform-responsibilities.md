# DFW Matchday Concierge — Platform Responsibilities Authority

## Status

This file assigns mission responsibilities across execution platforms.

It is a governing source-authority file.

## Responsibility Principle

No single platform owns the full mission.

Claude is only the site execution lane.

ChatGPT Mission Control owns source authority, logic packaging, phase control, and prompt creation.

The owner/operator owns final business decisions and field execution.

Other platforms own repo control, deployment, domain/DNS, booking intake, QR, analytics, sponsors, payments if approved, and content/data updates.

## Owner / Operator Responsibilities

The owner/operator is responsible for:

- Final business decisions.
- Final service lane approval.
- Final pricing or quote policy.
- Vehicle details.
- Driver details.
- Daily availability.
- Booking acceptance.
- Booking rejection.
- Alternate driver approval.
- Hotel concierge relationships.
- Field observations.
- Route decisions.
- Sponsor approval.
- Payment/deposit approval if used.
- Final publish approval.
- Actual ride fulfillment.
- Guest communication.
- Missed conversion follow-up.

Owner handoff artifacts include approved source-authority package, approved vehicle/driver details, approved route list, approved booking policy, approved sponsor policy, approved publish decision, and field feedback after deployment.

## ChatGPT Mission Control Responsibilities

ChatGPT Mission Control is responsible for:

- Creating source-authority files.
- Organizing the mission into phases.
- Creating Claude prompts.
- Preserving platform boundaries.
- Auditing logic.
- Identifying missing responsibilities.
- Creating QA checklists.
- Creating correction prompts.
- Reviewing Claude reports with the owner.
- Producing next-step recommendations.

ChatGPT Mission Control handoff artifacts include source-authority package, Claude placement prompt, Claude implementation prompts, QA checklist, responsibility matrix, handoff instructions, correction prompts, and issue register if needed.

## Claude Responsibilities

Claude is responsible for:

- Placing locked source-authority files exactly as provided only if instructed.
- Implementing assigned site phases only.
- Updating pages/components/copy when directly instructed.
- Preserving approved design standard.
- Performing in-app review.
- Reporting files changed.
- Reporting risks.
- Reporting unresolved issues.
- Reporting implementation limitations.

Claude must not create source authority, rewrite locked source files, summarize locked source files instead of placing them, redesign the site, expand scope, invent business rules, invent driver claims, invent service lanes, invent sponsor placements, deploy without explicit authorization, or claim external testing success without evidence.

## GitHub / Repository Responsibilities

GitHub or the active repository system is responsible for source control, file history, branch management, commit tracking, pull request review if used, diff review, rollback path, issue tracking if enabled, storing source-authority files, and storing site implementation files.

GitHub handoff artifacts include branch name, commit hash, pull request link if used, changed files list, diff summary, rollback note, and open issues.

## Hosting / Deployment Responsibilities

The hosting/deployment platform is responsible for preview deployment, production deployment only after owner approval, build logs, live URL, deployment status, redirect status, SSL status, and rollback if needed.

Hosting handoff artifacts include preview URL, production URL if approved, build log result, deployment errors if any, Reserve page URL, QR target test result, and mobile page test result.

## Domain / DNS Responsibilities

Domain/DNS platform is responsible for domain records, subdomain records, redirects, SSL routing, stable Reserve page URL, and QR target support.

Domain/DNS handoff artifacts include active domain, Reserve page URL, redirect rules, DNS status, and SSL status.

## Booking Intake Responsibilities

Booking intake system is responsible for ride request capture, Pickup + Return fields, airport return capture, passenger count, luggage count, primary ride type, conditional add-ons, alternate driver permission, limited availability acknowledgment, owner notification, guest confirmation, submission storage, and failure states.

Booking intake handoff artifacts include form URL, submission destination, notification recipient, test submission result, field list, required field validation, confirmation message, and failure state test.

## Content / Data Responsibilities

Content/data system is responsible for team pages, group standings, team schedules, completed match scores, match recaps, upcoming matchup previews, tournament news, tournament calendar, Fan Festival dates, watch events, sponsor inventory, availability notes, last updated timestamps, fallback states, and manual override process.

Content/data handoff artifacts include data source location, update owner, last updated field, fallback state, manual override process, and content refresh procedure.

## QR / Field Materials Responsibilities

QR/field materials system is responsible for QR code creation, QR target URL, print-safe QR size, field card creation, hotel concierge card creation, campaign source URLs, scan testing, and mobile reservation test.

QR/field materials handoff artifacts include QR target URL, QR image file, print file, campaign source parameter, scan test result, and mobile reservation test result.

## Analytics Responsibilities

Analytics system is responsible for QR scans, Reserve page visits, form starts, form submissions, source tracking, hotel concierge source tracking, airport source tracking, stadium source tracking, Fan Festival source tracking, Stockyards source tracking, Terry Black’s source tracking, and Katy Trail source tracking.

Analytics handoff artifacts include tracking platform, tracked events, UTM/source list, submission source capture status, dashboard link if available, and known tracking gaps.

## Payment / Deposit Responsibilities

Payment/deposit platform is responsible only if owner approves payment use.

Payment/deposit platform may handle deposit collection, quote payment, manual payment link, payment confirmation, refund/cancellation rules, and payment records.

Payment handoff artifacts include payment method, payment link rules, deposit language, refund/cancellation language, confirmation workflow, and owner approval status.

Do not add payment or deposit collection unless explicitly approved by the owner.

## Sponsor / Advertise Responsibilities

Sponsor system is responsible for sponsor inventory, sponsor inquiry intake, sponsor approval, sponsor asset collection, sponsor page updates, sponsor placement control, sponsor labeling, and avoiding official affiliation confusion.

Sponsor handoff artifacts include sponsor page URL, inventory table, sponsor inquiry form if used, approved sponsor list, ad asset files, placement dates, and approval status.

## Field Operations Responsibilities

Field operations are responsible for actual ride fulfillment, availability windows, guest communication, confirmed ride management, waitlist, airport route decisions, stadium route decisions, Fan Festival route decisions, Dallas / Fort Worth / Arlington route decisions, matchday staging, hotel concierge opportunities, alternate driver handoff, revenue tracking, missed conversion tracking, and follow-up.

Field operations handoff artifacts include daily availability, confirmed bookings, waitlist, open windows, concierge opportunities, missed conversion notes, follow-up list, and revenue notes.

## Responsibility Boundary Summary

Owner/operator owns final decisions and field execution.

ChatGPT Mission Control owns source authority and phased reasoning.

Claude owns site implementation only.

GitHub owns source history.

Hosting owns deployment.

Domain/DNS owns stable URL routing.

Booking intake owns request capture.

QR/field materials own real-world scan-to-reserve flow.

Analytics owns visibility into behavior.

Payment owns collection only if approved.

Sponsors own controlled advertising only after owner approval.

Field operations own actual revenue execution.
