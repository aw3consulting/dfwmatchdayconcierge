# DFW Homepage Section-by-Section Premium Website Builder Governance Standard v1.0.0

## Controlled Action

`dfw_homepage_section_by_section_premium_website_builder_governance_standard`

## Status

`LOCKED / SECTION-BY-SECTION GOVERNANCE CREATED / FULL HOMEPAGE ONE-PASS EXECUTION BLOCKED / SECTION SKELETON REQUIRED BEFORE CONTENT BUILD`

## Purpose

This standard converts homepage execution from one-pass page generation into a premium website-builder workflow.

The homepage must be defined, rendered, reviewed, revised, and approved section by section before a complete homepage build is authorized.

## Core Determination

The v1.0.4 failure proves that governance can still fail when the acting session is allowed to build the whole homepage at once.

The new rule is:

`No homepage may be generated as a full page until each section has been independently defined, rendered with CSS, reviewed, and approved.`

## Approved Section Units

The homepage is now governed by these required section units only:

1. Header
2. Navigation
3. Hero
4. Section 1
5. Section 2
6. Section 3
7. Section 4
8. Footer

Each section must be treated as an independent build unit.

## Section-First Build Rule

Each section must pass through this sequence:

1. Section definition
2. Section option selection
3. Section skeleton render
4. Section CSS review
5. Section mobile behavior review
6. User approval
7. Content authorization
8. Final content insertion

No section may receive final content before its skeleton and CSS behavior are approved.

## Skeleton Content Rule

The first rendered version of each section may contain only:

1. section name
2. section tagline
3. structural container
4. visual layout treatment
5. spacing behavior
6. responsive behavior

The first rendered version may not contain:

1. final copy
2. service cards
3. sponsor details
4. route links
5. CTAs beyond placeholder labels if needed for layout
6. icons beyond neutral structural placeholders
7. official-source language
8. provider handoff language
9. partner names or logos
10. operational claims

## Premium Website Builder Model

Each section must be governed like a premium website builder module.

For each section, the acting session must present:

1. section purpose
2. section visual role
3. section layout options
4. recommended option
5. desktop behavior
6. mobile behavior
7. allowed content type
8. blocked content type
9. CSS requirements
10. approval status

## Section Option Requirement

Before a section is coded, the acting session must identify at least three viable premium website section options and select one recommended option.

The section options must be real website layout patterns, such as:

1. minimal luxury bar
2. split hero
3. editorial hero
4. card rail
5. feature strip
6. sponsor band
7. trust strip
8. compact footer

The acting session must not skip option analysis.

## Approval Requirement

Each section requires explicit user approval before the next section may be finalized.

The allowed approval states are:

1. Not Started
2. Defined
3. Skeleton Rendered
4. Under Review
5. Revision Required
6. Approved
7. Locked

No section may move to Locked unless the user approves it.

## Full Homepage Composition Rule

The complete homepage may only be assembled after all eight section units are locked.

Until then, the working artifact must remain a section skeleton or a section-by-section review surface.

## Failure Triggers

The execution fails if:

1. the acting session builds the full homepage before section approval
2. final content is inserted before skeleton approval
3. any section lacks a section name and tagline in the skeleton stage
4. CSS is reviewed only at whole-page level
5. mobile behavior is skipped for any section
6. section options are not presented
7. user approval is assumed
8. a section carries content belonging to another section
9. a section repeats a concept already assigned elsewhere
10. the prototype becomes a complete content page before all sections are approved

## Final Determination
