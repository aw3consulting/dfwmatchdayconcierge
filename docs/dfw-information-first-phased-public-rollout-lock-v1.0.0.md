# DFW Information-First Phased Public Rollout Lock v1.0.0

## 1. Locked Directive

DFW Matchday Concierge will launch first as DFW Matchday Information + Planning Command Hub through a phased public rollout: Launch Option B countdown and information preview, followed by Launch Option C single-page live public gateway, followed by Launch Option A full information gateway with gated paid planning, client hubs, monetization, API-ready infrastructure, and internal-only operations.

## 2. Brand and Product Structure

Brand:

`DFW Matchday Concierge`

Public-facing product layer:

`DFW Matchday Information + Planning Command Hub`

Paid planning product:

`Custom Matchday Plan + Client Hub Access`

Premium support layer:

`Operator-reviewed VIP / Team Family / Media-Corporate support`

Internal system:

`Driver, Concierge, Fulfillment, Assignment, Escalation, and Closeout operations`

## 3. Active Phased Launch Path

### 3.1 Launch Option B

`Countdown + Information Preview`

Purpose:

Go live professionally without overpromising capacity.

Includes:

- premium landing page
- launch countdown
- information-first positioning
- public guide preview
- Custom Matchday Plan waitlist
- limited manual inquiry interest
- clear phased-launch language
- no unsupported claims

### 3.2 Launch Option C

`Single-Page Live Public Gateway`

Purpose:

Turn the launch page into a useful public information gateway.

Includes:

- public information categories
- guide previews
- custom planning CTA
- paid Client Hub CTA
- information-only response path
- referral option
- escalation option
- no guarantee boundary

### 3.3 Launch Option A

`Full Information Gateway Launch`

Purpose:

Public scale, paid conversion, premium fulfillment, and controlled operations.

Includes:

- full public information hub
- SEO-ready public content sections
- monetization-ready ad/referral/sponsored zones
- API-ready infrastructure
- Custom Matchday Plan product
- paid Client Hub access
- client-specific hubs
- complaint / information correction flow
- emergency / safety / medical information page
- legal / privacy / terms
- internal-only operations

## 4. Rejected and Fallback Options

Rejected:

`Concierge-first full booking launch`

Reason:

This would overexpose contact, capacity, fulfillment, driver/concierge expectations, complaints, and liability.

Emergency-only fallback:

`Private beta / early access page`

Use only if public launch risk becomes unacceptable.

## 5. Public User Journey

The public user journey is now:

`Information Gateway -> Public Guide -> Custom Matchday Plan CTA -> Paid Client Hub -> Operator Review -> Quote / Scope -> Confirmation -> Fulfillment`

## 6. Layer Separation

Public layer:

- information
- guides
- SEO
- ads/referrals where approved
- information issue reporting
- custom planning CTA

Paid planning layer:

- Custom Matchday Plan
- paid Client Hub access
- saved itinerary
- status board
- limited support

Qualified premium layer:

- VIP
- Team Family
- Media / Corporate
- direct bookings
- quotes
- operator review

Internal operations layer:

- Driver Hub
- Concierge Hub
- Fulfillment Queue
- Assignment Board
- Escalation Board
- Closeout Board

Infrastructure layer:

- manual tracker first
- backend database later
- Oracle OCI future command center
- DreamHost static public site after approval

## 7. Contact Center Rule

Public contact access is no longer assumed as the default full public layer.

Direct contact center access should be gated through:

- paid Client Hub layer
- verified Client Hub layer
- controlled manual intake where explicitly approved

The public layer should prioritize information, custom planning CTA, waitlist, and issue/correction reporting.

## 8. Driver and Concierge Rule

Driver and Concierge Hubs are internal-only operating surfaces.

Public pages may include Driver Interest and Concierge Interest only.

Client pages may display driver or concierge status only inside a paid, verified, or confirmed client-specific hub.

## 9. Monetization Rule

The monetization hierarchy is:

1. public information monetization
2. API / data / partner integration monetization
3. nominal Custom Matchday Planning fee
4. paid Client Hub access
5. direct booking and fulfillment

## 10. Production Standing

This lock does not authorize:

- DreamHost production deployment
- DNS/domain routing changes
- final QR activation
- PR merge to `main`
- public production launch
- live AI automation
- live payment collection
- live dispatch
- live routing automation
- automatic driver assignment
- automatic concierge assignment
- official affiliation claims
- fulfillment guarantees

## 11. Final Determination

This file is the governing phased-rollout posture for the next scope, mockup, frontend, launch, and production-readiness decisions.
