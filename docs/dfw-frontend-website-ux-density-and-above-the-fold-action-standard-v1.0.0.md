## 3. Purpose

This addendum hardens the DFW Matchday Information + Planning Hub mockup standard by adding website UX density, above-the-fold action, browser-interface realism, and speed-to-information controls.

It extends:

`docs/dfw-frontend-mockup-pdf-generation-pdf-to-html-conversion-and-responsiveness-standard-v1.0.0.md`

The prior frontend standard governs viewport, full-page scroll format, PDF generation, semantic HTML conversion, responsive behavior, typography, spacing, accessibility, sponsor inventory layout, JavaScript boundaries, and claim controls.

This addendum adds a required website interaction layer so mockups function like high-conversion web pages, rather than static report pages or formatted documents.

## 6. Core UX Determination

Every DFW public page mockup must behave as a website interface first.

No homepage or linked page may read primarily as a document, report, sales brief, or static explanation page.

The first visible viewport must operate as a command gateway that lets visitors quickly choose a planning path, guide path, official-source path, sponsor path, or appropriate reviewed-support path.

The page must prioritize fast action, clear routing, visible utility, and conversion-ready browser components.

## 7. Above-the-Fold Action Hub Requirement

Every homepage and high-value linked page must include a functional action hub in the first visible desktop viewport.

The action hub must include, where relevant:

1. Find My Matchday Plan search or selector
2. Build My Matchday Plan CTA
3. Popular Needs route cluster
4. quick access to major guide categories
5. language control
6. contrast control
7. title sponsor or page sponsor inventory lane
8. Advertise With Us business CTA
9. clear reviewed-support boundary only as secondary support

The first desktop viewport must allow a visitor to reach at least five high-value planning paths without scrolling.

The first mobile viewport must allow a visitor to reach the primary planning action and at least three high-value planning paths with minimal scrolling.

## 8. Popular Needs Requirement

Every homepage and visitor-facing linked page must include a Popular Needs or equivalent fast-path route cluster near the top of the page.

Popular Needs may include, as page context requires:

1. Airport Arrivals
2. Stadium Route
3. Transportation
4. Dining and Watch Parties
5. Hotels and Areas
6. Team Hubs
7. Parking and Shuttles
8. Official Sources
9. Luggage Support
10. Wellness and Recovery
11. Fan Fest and Events
12. Sponsor Options

Popular Needs must be designed as tappable or clickable interface elements, not paragraph text.

## 9. Desktop UX Density Standard

Desktop mockups must use the 1440 CSS px viewport and 1320px centered content container from the frontend standard, while also meeting these UX density controls:

1. the first viewport must be visibly interactive
2. the hero must include utility, not just branding
3. the plan builder must appear as an actionable module
4. major guide paths must appear in compact card or pill form near the top
5. sponsor inventory must be visible but not consume the page as an ad document
6. official-source access must be discoverable without deep scrolling
7. provider or referral pathways must appear as interface cards, not long copy blocks
8. reviewed support must remain secondary and restricted by audience boundary
9. dead zones must be minimized
10. premium negative space is allowed only when it improves visual hierarchy or interaction clarity
11. no large empty dark area may exist without a defined visual, routing, or interaction purpose
12. multi-column utility layouts should be used where they improve speed to information

## 10. Mobile UX Density Standard

Mobile mockups must use the 390 CSS px viewport and full-page scroll behavior from the frontend standard, while also meeting these tap-first controls:

1. the first mobile viewport must prioritize action over explanation
2. Find My Matchday Plan must be immediately visible or reachable through the first action module
3. Build My Matchday Plan must appear before long explanatory content
4. Popular Needs cards or chips must appear near the top
5. guide categories must be compact and tappable
6. sponsor inventory must be readable and tappable without becoming a long ad wall
7. official-source access must be compact and clear
8. provider or referral options must appear as concise cards
9. reviewed support must remain secondary
10. paragraph-heavy sections must be converted into compact disclosure, cards, accordions, steps, or chips
11. mobile layouts must avoid long briefing-style runs before utility appears
12. no horizontal scrolling may be required

## 11. Document-Like Layout Prohibition

A page fails if it reads primarily as a static document.

Document-like failure indicators include:

1. long paragraph blocks before core actions
2. large passive explanation sections
3. weak action hierarchy above the fold
4. guide paths hidden below excessive copy
5. sponsor inventory presented like a report section rather than actionable inventory
6. official-source links presented only as plain text
7. provider paths presented only as descriptive copy
8. large empty dark zones with no action purpose
9. page sections that feel like a PDF brief rather than reusable web components
10. footer or trust content dominating before primary visitor utility is established

## 12. Content Conversion Requirement

Paragraph-heavy content must be converted into web-native modules.

Approved content forms include:

1. cards
2. route tiles
3. action chips
4. form selectors
5. tabs where appropriate
6. accordions where appropriate
7. step modules
8. compact comparison blocks
9. status badges
10. sponsor inventory cards
11. official-source cards
12. provider handoff cards
13. reviewed-support boundary panels

Long explanatory paragraphs may remain only where they support trust, disclaimer, or necessary context and do not block the primary action flow.

## 13. First Viewport Minimum Content Standard

The homepage desktop first viewport must include:

1. approved DFW logo and brand header
2. title sponsor availability lane
3. Advertise With Us CTA
4. stable navigation labels
5. homepage hero value statement
6. Find My Matchday Plan action module
7. Build My Matchday Plan CTA
8. Popular Needs cluster

10. language and contrast controls
11. no official affiliation implication

The homepage mobile first viewport must include:

1. approved DFW logo and brand header
2. menu access
3. compact language and contrast access
4. homepage hero value statement
5. Find My Matchday Plan action module or immediate shortcut
6. Build My Matchday Plan CTA
7. Popular Needs cluster or first visible portion of it
8. no QR-dominant action
9. no Connect-as-general-visitor CTA

## 14. Sponsor Inventory UX Standard

Sponsor inventory must remain visible, commercial, and actionable without making the page feel like an ad sheet.

Sponsor inventory must:

1. show available status until sold and approved
2. use compact cards or lanes
3. clearly identify placement type
4. avoid real sponsor logos unless approved
5. avoid sold-looking treatments unless sold and approved
6. avoid official sponsor implication
7. offer advertiser CTAs such as Advertise With Us, View Sponsor Options, Reserve This Slot, Sponsor This Guide, or View Available Inventory
8. stay secondary to the visitor utility flow on general visitor pages

## 15. Official-Source and Provider Path UX Standard

Official-source access must be visibly distinct from platform content.

Provider, referral, and partner paths must be clearly presented as third-party or handoff options where applicable.

The layout must avoid implying that DFW Matchday Concierge is an official tournament, team, venue, airport, city, or sponsor channel.

Official-source and provider sections should use cards, labels, chips, or structured link groups rather than long explanation blocks.

## 16. Premium Review Boundary UX Standard

Premium reviewed support must remain controlled.

General visitors must be served primarily through planning tools, guides, official-source paths, sponsor/partner options, referral/provider paths, and compact self-serve flows.

Reviewed support may appear only as a secondary fallback and must be framed for:

1. VIP needs
2. Team support needs
3. Media or corporate coordinator needs
4. complex group needs
5. complex arrival, routing, or planning scenarios

No public visitor page may imply backend Command Hub access for general fans.

## 17. Browser Realism Standard

PDF mockups must represent actual browser components that can translate into semantic HTML and responsive CSS.

Browser realism requires:

1. realistic header behavior
2. realistic navigation density
3. realistic card grids
4. realistic form modules
5. realistic responsive stacking
6. realistic spacing and gutters
7. realistic CTA sizing
8. realistic section sequencing
9. no static presentation-board composition
10. no artificial canvas layout that cannot become HTML/CSS
11. no fixed-height text containers that would clip during real content expansion
12. no absolute-positioned text-bearing blocks unless explicitly justified and conversion-safe

## 18. Reading Order and Conversion Semantics

Mockups must support future PDF-to-HTML conversion.

The visible page order should map cleanly to semantic HTML:

1. skip link
2. header
3. navigation
4. main hero
5. primary action hub
6. Popular Needs
7. guide gateway
8. sponsor inventory
9. official-source section
10. provider/referral section
11. premium review boundary
12. trust/footer
13. disclaimer

Visual design may use multi-column layouts, but the semantic content order must remain logical for conversion.

Extraction or reading-order anomalies must be reviewed if they suggest footer, disclaimer, or lower-priority content is structurally placed before primary page content.

## 22. Automatic Failure Triggers

A mockup fails under this addendum if it contains:

1. first viewport without a functional action hub
2. visitor planning actions hidden below passive explanation
3. document-like layout as the dominant page impression
4. excessive paragraph density before primary actions
5. large empty dark zones without visual or interaction purpose
6. missing Popular Needs or equivalent fast-path route cluster
7. sponsor inventory missing or presented as sold without approval
8. official-source access missing where relevant
9. provider/referral access missing where relevant
10. premium reviewed support presented as the primary general visitor path
11. Connect used as a general visitor CTA
12. QR presented as a dominant on-site CTA
13. backend Command Hub access implied for general visitors
14. unsupported official affiliation claim
15. logo drift from the approved DFW logo direction
16. unreadable microtext
17. clipped text
18. overlapping text
19. mobile layout that reads as a long briefing page before actions appear
20. desktop layout that feels like a static PDF report rather than a browser interface
