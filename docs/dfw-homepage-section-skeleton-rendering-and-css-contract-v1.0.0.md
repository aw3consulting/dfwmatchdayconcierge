## Purpose

This contract defines how the next homepage prototype must be rendered before any final homepage content is inserted.

The next execution must prove structure, spacing, hierarchy, CSS quality, and responsive behavior using section names and section taglines only.

## Core Rule

The next homepage prototype may not begin as a content homepage.

It must begin as a governed section skeleton.

The only content allowed in each section during the skeleton stage is:

1. section name
2. section tagline
3. neutral layout labels if required for accessibility or review

The skeleton must render exactly these sections:

4. Section 1

## CSS Rendering Requirements

The skeleton must render to real CSS, not an image mockup.

The CSS must demonstrate:

1. global token usage
2. controlled container width
3. premium dark visual system
4. section spacing
5. desktop hierarchy
6. mobile reflow
7. header behavior
8. navigation behavior
9. hero scale
10. section rhythm
11. footer containment
12. no overlap
13. no clipped text
14. no horizontal mobile scroll

## Website Builder Options Requirement

Before skeleton code execution, each section must present premium website-builder options.

Each section must include at least three options from relevant website patterns.

Examples:

1. luxury minimal bar
2. split-screen hero
3. editorial hero
4. card rail
5. feature strip
6. sponsor band
7. partner recognition strip
8. trust rail
9. compact footer
10. stacked mobile panel

The acting session must recommend one option for each section and explain why it fits the DFW Matchday Concierge homepage.

## Section Skeleton Approval Rule

Each section must move through these states:

1. Defined
2. Options Presented
3. Recommended Option Selected
4. Skeleton Rendered
5. Under Review
6. Revision Required or Approved
7. Locked

No final content may be inserted into a section before the section is locked.

## File Authority for Skeleton Stage

The skeleton stage may create or edit only:

1. `docs/prototypes/homepage-section-skeleton-v1.0.0/review.html`
2. `docs/prototypes/homepage-section-skeleton-v1.0.0/assets/css/tokens.css`
3. `docs/prototypes/homepage-section-skeleton-v1.0.0/assets/css/section-skeleton.css`
4. `docs/prototypes/homepage-section-skeleton-v1.0.0/assets/css/preview-fixes.css` if needed
5. `docs/prototypes/homepage-section-skeleton-v1.0.0/README.md` if needed

The skeleton stage may not edit:

1. `docs/prototypes/homepage-v1.0.4/`
2. production files
3. linked pages
4. PDF files
5. image files
6. deployment files

## Review Output Requirement

After skeleton render, the acting session must return:

1. section list
2. section option table
3. selected option per section
4. files created or edited
5. commit SHA
6. preview URL
7. desktop review readiness
8. mobile review readiness
9. explicit statement that final content remains blocked

## Failure Triggers

The skeleton execution fails if:

1. it includes final marketing copy
2. it includes final cards or sponsor details
3. it includes route links beyond structural placeholders
4. it skips option analysis
5. it renders a full content homepage
6. it combines sections without clear section boundaries
7. it fails to render section names and taglines
8. it edits the v1.0.4 full homepage prototype directly
9. it proceeds to content insertion without user approval
10. it claims approval without user approval
