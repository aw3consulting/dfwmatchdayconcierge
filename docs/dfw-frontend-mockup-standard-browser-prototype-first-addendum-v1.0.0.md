## 3. Purpose

This addendum role-qualifies future DFW Matchday Information + Planning Hub visual development work and installs the browser-prototype-first rule.

It prevents recurrence of the composite-board failure, the v1.0.1 document-like drift, and the v1.0.2 dashboard-overcompression failure.

## 6. Core Rule

Future homepage and linked-page mockups must be generated from browser-rendered HTML/CSS prototypes, not image-style impressions, visual boards, prompt-generated webpage images, compressed design posters, static mockup boards, shrink-to-fit PDF exports, or dashboard-style visual approximations.

The approved v1.0.0 homepage mockup remains the positive concept authority. The v1.0.1 and v1.0.2 outputs are rejected negative controls only. Future work must preserve the approved v1.0.0 concept while satisfying the frontend standard and browser-prototype-first conversion controls.

## 7. Browser Prototype First Requirement

Before homepage v1.0.3 or linked-page PDF output is created, a browser-rendered HTML/CSS prototype must exist for the target page.

The prototype must include the semantic page shell, approved header, stable navigation, sponsor lane, hero, main action module, Popular Needs route cluster, guide gateway, sponsor inventory, source lane, provider/referral handoff, reviewed support boundary, trust row, footer, disclaimer, responsive behavior, tokenized CSS, accessible controls, and claim-safe copy.

## 8. Homepage Preservation Matrix Requirement

Future homepage v1.0.3 work must satisfy:

`docs/dfw-homepage-preservation-matrix-v1.0.0.md`

The preservation matrix is a mandatory gate before visual output acceptance.

## 9. Native Capture Rules

All future PDF review artifacts must be captured from browser-rendered prototypes.

Required capture rules:

1. desktop capture uses native 1440 CSS px viewport
2. desktop content uses 1320px centered max-width unless superseded by a later locked standard
3. mobile capture uses native 390 CSS px viewport
4. shrink-to-fit PDF export is blocked
5. image-board export is blocked
6. wide canvas poster export is blocked
7. phone mockup inside desktop board is blocked
8. multi-page artifact inside one output file is blocked
9. static screenshot collage is blocked
10. output that hides actual browser stacking or reflow behavior is blocked

## 10. Semantic Read-Order Rule

Future PDF outputs must preserve this logical order for conversion:

1. skip link
2. header
3. navigation
4. hero
5. main action module
6. guide gateway
7. sponsor inventory
8. source section
9. provider handoff
10. reviewed support boundary
11. trust row
12. footer
13. disclaimer

Any output whose extracted text begins in lower-priority page content fails until the prototype structure is corrected.

## 11. UX Density Balance Rule

Future work must balance utility density with premium public website quality.

The layout must avoid both failure extremes:

1. document-like scroll output with passive copy, dead zones, and slow utility
2. overcompressed dashboard output with cramped cards, microtext, weakened brand presence, and reduced premium feel

The approved direction is a premium browser interface with clear hierarchy, fast action, purposeful spacing, scannable components, and conversion-ready structure.

## 12. Sponsor Commercial Value Rule

Sponsor inventory must remain commercially valuable and premium.

Sponsor treatment must include clear available status, page-context placement label, advertiser CTA, premium visual treatment, claim-safe language, approved mark discipline, sale-status discipline, and no devaluation through tiny cards, buried placement, or unclear inventory language.

## 13. Automatic Failure Triggers

Future work fails if it repeats any prior failure class, including board output, composite batch output, non-prototype visual impression, shrink-to-fit export, semantic read-order failure, document-like layout, dashboard overcompression, microtext, clipped text, overlapping text, logo drift, brand naming drift, missing sponsor lane, missing advertiser CTA, missing available inventory, restricted claim drift, dominant on-site QR action, or restricted backend/public visitor drift.

## 14. Required Recovery Sequence

The recovery sequence is:

1. verify source authority on `main`
2. commit and verify this addendum on `main`
3. commit and verify the homepage preservation matrix on `main`

8. review the prototype against the preservation matrix and section map
9. export homepage desktop and mobile PDFs only from the approved browser prototype
10. approve or reject the homepage pair
11. proceed to linked pages only after homepage approval
12. commit approved mockups only after explicit user authorization
