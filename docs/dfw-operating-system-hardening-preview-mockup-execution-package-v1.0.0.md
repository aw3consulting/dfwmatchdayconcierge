## 4. Mockup Product Thesis

The mockup must prove one thing immediately:

`DFW Matchday Concierge is a World Cup matchday operating system for Dallas / Fort Worth visitors, not a ride booking page.`

The experience must communicate:

- planning before fulfillment
- lanes before services
- hub activation before personal itinerary depth
- operator review before confirmation
- direct fulfillment only when accepted
- referrals, affiliates, API routing, and ads as separate monetization lanes
- premium hub privacy for VIP, Team Family, Media, and Corporate users
- scalable fan guidance for public traffic

The visual language should blend:

- premium concierge

## 22. Acceptance Checklist

The mockup passes only if:

- homepage reads as operating system, not transportation site
- four lanes are visible above the fold on desktop
- mobile inquiry path is reachable within three taps
- VIP and Team Family are visually privacy-protected
- Media / Corporate is quote-ready and operational
- Fan hub can scale through free guidance and activated planning
- ad/referral zones are restricted to appropriate public fan contexts
- every service claim has a status boundary
- Tahoe is presented only as approved VIP standard, not guaranteed
- AI intake is shown as planning preview, not booking confirmation
- map does not imply live routing or live dispatch
- payment is shown as operator-confirmed
- DreamHost production remains blocked

## 23. Rejection Triggers

Reject the mockup if:

- it looks like a generic black luxury transportation site
- it makes VIP look like a single ride product
- it implies guaranteed Tahoe access
- it implies official tournament affiliation
- it places ads inside premium private hubs
- it shows fake country/team/tournament marks
- it uses fake or AI-artifact photography
- it suggests live dispatch before implementation
- it suggests live payment automation before implementation
- it omits operator review from confirmation language
- it buries inquiry action on mobile
