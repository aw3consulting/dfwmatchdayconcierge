## Purpose

This gate prevents the PDF replica workflow from accepting a visual approximation that is built on poor, invalid, inaccessible, or unmaintainable HTML/CSS.

A replica is not valid unless the code underneath it is valid, responsive, maintainable, and production-convertible.

## Non-Negotiable Rule

No PDF-to-HTML replica may pass if it produces poor or invalid HTML/CSS code.

Visual similarity cannot override:

1. Valid HTML.
2. Valid CSS.
3. Semantic structure.
4. Maintainable code organization.
5. Responsive behavior.
6. Accessibility basics.
7. Production-safe copy and claim boundaries.
8. Browser proof.

## Automatic Failure Conditions

The replica fails even if it looks closer to the PDF when any of the following are true:

1. Invalid HTML structure.
2. Broken tags or malformed nesting.
3. Unlabeled or inaccessible form controls.
4. CSS that depends on brittle one-off hacks without documentation.
5. Desktop-only layout.
6. Mobile horizontal overflow.
7. Hidden required sections on mobile without approval.
8. Unreviewed claim or metric copy in production scope.
9. Public route replacement before validation.
10. No desktop and mobile browser evidence.

## Code Quality Hardening Requirements

Before production route replacement, the isolated prototype must be hardened through:

1. Readable, indented HTML.
2. Section comments matching the PDF-defined regions.
3. Semantic landmarks and clear heading order.
4. Valid form labels and accessible controls.
5. CSS organized into layers:
   - tokens
   - base
   - layout
   - components
   - responsive
   - utilities
6. Expanded design tokens for spacing, typography, panel, border, shadow, z-index, and breakpoints.
7. Consistent class naming by component purpose.
8. Focus states for links, buttons, selects, and inputs.
9. Responsive testing at 390px, 430px, 768px, 1024px, and desktop width.
10. No production claim leakage from the prototype reference layer.

## Responsive Frontline Requirements

Responsive review is not a final polish item. It is a frontline acceptance gate.

The build must preserve:

1. Clean mobile header and primary action.
2. Clean mobile hero readability.
3. Tap-friendly inquiry stack.
4. Readable mobile operating paths.
5. Map/routes converted into mobile-readable structure.
6. Media/corporate module legibility.
7. Team hub stacking and benefit readability.
8. Driver, concierge, fulfillment, status, insight, and footer presence.
9. No horizontal overflow.
10. No content loss without explicit approval.
