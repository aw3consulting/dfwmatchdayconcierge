## 14. Asset Handling Rules

The frontend shell must use:

- approved DFW Matchday Concierge logo only
- schematic placeholders where approved photography is unavailable
- no fake official marks
- no fake team marks
- no fake country marks beyond plain text references
- no unverified Tahoe image
- no unapproved DFW photography
- no unlicensed celebrity, player, federation, or team imagery

If final assets are unavailable, use controlled schematic panels and clear placeholders.

Required disclosure labels:

- Direct DFW Service

- tab switching

- chip selection visuals

- non-submitting form-shell behavior

It may not support:

- real payment submission
- user account creation
- authentication
- live AI calls
- live dispatch calls
- live map routing calls
- third-party API calls
- storage of sensitive user data

Preferred commit sequence:

1. data/status badge support
2. shared shell CSS and static UI helpers
3. homepage shell
4. connect/QR shell

6. Media / Corporate and Fan Hub shells

## 22. Stop Conditions

Stop implementation and return to governance review if:

- a required claim cannot be mapped
- a frontend requirement implies live automation
- a page requires unapproved imagery
- a page implies official affiliation
- a payment/deposit action appears live
- a map appears live
- VIP appears as a single-trip ride product
- Fan Hub receives default VIP/Tahoe positioning
- ads appear in restricted private contexts
- DreamHost deployment becomes implied or requested without explicit approval

## 23. Final Opening Determination

The bounded frontend shell implementation lane is opened.

Scope is limited to static review-surface frontend shell implementation.

GitHub Pages may remain a temporary review surface.

DreamHost production deployment remains blocked.

Final launch remains blocked.

Live AI, payment, dispatch, routing, authentication, official affiliation, and fulfillment automation remain blocked.
